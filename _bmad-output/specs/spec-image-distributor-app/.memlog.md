- (decision) Why: friend manually sends daily images to 8-10 paying WhatsApp/email subscribers, costing him significant time and risking visible repeats that draw complaints; building a private sideloaded automation app for this, licensed independently of the friend-customer payment relationship
- (capability) CAP-1 Image library management: intent - operator uploads images and flags each active/inactive; success - an inactive image is never selected for sending, reactivating makes it selectable again
- (capability) CAP-2 Receiver configuration: intent - operator configures up to ~10 receivers, each with name, +91 mobile number, delivery channel (WhatsApp or email), per-receiver min/max daily image count, and daily send schedule; success - each receiver sends per its own schedule/channel/count range, not a shared global setting
- (capability) CAP-3 Daily randomized selection per receiver: intent - at each scheduled send, pick a random count between that receiver's min/max, then that many active images, avoiding repeats sent to that receiver in the last 7 days unless the eligible pool is too small to cover the count; success - two sends to the same receiver differ in count and image set; no receiver gets an image already sent to them within 7 days unless the active-and-unsent pool was smaller than the needed count, in which case repeats are allowed
- (capability) CAP-4 Offline-safe delivery queue: intent - selected sends are queued by image ID (not image data) and delivered via WhatsApp/email; failures retry automatically and resume once connectivity returns; success - a send that fails for lack of network retries up to 3 times and completes once connectivity is restored with no operator action; if the tablet is offline through an entire scheduled window, only that day's batch sends on reconnect, earlier missed days are not backfilled
- (capability) CAP-5 Delivery dashboard / proof of delivery: intent - operator views, per receiver, a log of what was sent and when, covering at least the last 30 days; success - for any receiver and day in the last 30 days, dashboard shows the timestamp of each image sent, sufficient to resolve a customer 'I did not get my images' dispute
- (capability) CAP-6 First-run customer registration: intent - on first launch the app collects the installer's ('customer') name and city once, locks them permanently, and registers the install with an external admin system; success - name/city cannot be edited after first entry; the same values are POSTed to the admin registration endpoint at first-run time
- (capability) CAP-7 Compliance/licensing gate: intent - app checks a hardcoded compliance API using the registered name+city and halts only on an explicit non-compliant response, continuing to operate otherwise; success - an explicit non-compliant response halts all sending immediately with a contact-admin message; an unreachable/failing API call does not halt sending, it retries next scheduled check while sends continue normally
- (constraint) App data persists in a local PostgreSQL database only; no cloud database dependency
- (constraint) App is distributed by direct package transfer (sideloaded) to the installer, never via Play Store/App Store; no store-based distribution or update mechanism
- (constraint) Receiver mobile numbers assume a fixed +91 country code
- (constraint) Compliance API endpoint is hardcoded into the app binary, not user-configurable
- (constraint) Once an image is selected and queued for a send it is still delivered even if later flagged inactive before the queue fires; inactive only blocks future selection
- (constraint) Fail-open design principle: only an explicit non-compliant response may halt sending; every other failure (compliance API unreachable/timeout, active-image pool smaller than chosen count, 7-day no-repeat pool exhausted) must degrade gracefully rather than block
- (constraint) Failed transmissions retry a maximum of 3 times
- (decision) Non-goal: payment/billing between the operator (friend) and his receivers/customers is handled entirely outside this app
- (decision) Non-goal: no admin interface for managing compliance status within this app; that is done in a separate external system, manually and offline
- (decision) Non-goal: the app does not generate, edit, or curate images; it only distributes images the operator uploads
- (decision) Non-goal: no Play Store/App Store distribution or store-based update mechanism
- (assumption) Assumed target platform is Android, since direct APK-style sideload distribution without an app store is impractical on iOS without enterprise provisioning
- (assumption) Assumed +91 is fixed/hardcoded for all receivers, not a per-receiver selectable country code
