- (technique) started Yes And Building
- (idea) Compliance API takes customer name + city, returns compliant status (true/false, or richer response TBD)
- (idea) No expiry date on compliance status; value is set/managed in a separate external app's database, checked live over API
- (idea) Compliance result cached locally for 2 days before requiring a fresh check
- (idea) If online, always call API for fresh compliance check; only fall back to the 2-day cache when offline
- (idea) On non-compliant result, app halts entirely at that screen with a 'contact admin' message; nothing else changes/breaks
- (idea) Name/city entered at first-run can be edited later if the customer wants to correct it
- (idea) Revised: lock customer name+city permanently after first entry; simultaneously POST that info to admin's own database to register the install
- (idea) Install registers as compliant by default; admin manually flips to non-compliant separately/offline from this app when needed
- (technique) started Job to Be Done
- (idea by user) Underlying job: automate a manual daily image-sending task that currently eats a lot of his time
- (idea) Requirement surfaced from the manual process: images must not repeat immediately for the same receiver
- (idea) Receivers notice/complain if they get repeat images too often — variety matters to keep them satisfied
- (idea) Receivers are paying customers, not internal contacts — this is a subscription content business, not a personal broadcast tool
- (insight) Reframe: real job = reliably fulfill a paid daily-image subscription automatically with enough variety that paying customers don't churn/complain — the compliance hook may be less about the receiver's location and more about the app operator's own standing (e.g. license/payment) with whoever runs the compliance API
- (idea) Dashboard's real job: proof of delivery he can point to if a paying customer disputes 'I didn't get my images'
- (idea) Proof of delivery = timestamp per image sent to each receiver, logged and viewable in dashboard
- (idea) Min/max image count range is configurable per-receiver, not a single global setting — different customers can have different volume tiers
- (technique) started Chaos Engineering
- (idea) Survives chaos: on reconnect after an outage, only resend that day's missed batch — do not backfill prior missed days (he handles those manually)
- (idea) Once an image is selected and queued, it still gets sent even if later flagged inactive before the queue fires — inactive only affects future selection, not what's already queued
- (idea) If active image count is less than the random Z chosen, just send however many are active (no duplicates, no topping up)
- (idea) Revised rule (supersedes 2-day cache): sending only stops on an explicit non-compliant response; if the compliance API call itself fails/times out, just retry on the next scheduled run and keep sending in the meantime
- (idea) No-repeat mechanism: check the transmission queue/history for that receiver before picking, so an image already sent to them isn't picked again
- (idea) No-repeat window is 7 days per receiver; if the active image pool is too small to satisfy that mathematically, repeats are allowed as a fallback
- (idea) Retry failed transmissions 3 times regardless of failure reason; correctness of the receiver's number is the friend's own responsibility to get right
- (decision) Out of scope: payment/billing between friend and his customers is handled entirely offline, not by this app
- (insight) Synthesis: (1) compliance hook doubles as a remote install-licensing switch for the sideloaded app itself, independent of friend-customer payments which are out of scope; (2) proof-of-delivery, 7-day no-repeat check, and the 30-day dashboard are all views over one transmissions ledger (receiver_id, image_id, sent_at, status) in Postgres, not three separate features; (3) consistent 'fail open, not closed' design principle across every chaos answer — only an explicit non-compliant response is allowed to stop sending; everything else (short image pool, unreachable compliance API, exhausted no-repeat pool) degrades gracefully instead of blocking. Chosen direction: proceed with app as a subscription-fulfillment tool with proof-of-delivery, per-receiver tiers, and licensing-style compliance gate.
