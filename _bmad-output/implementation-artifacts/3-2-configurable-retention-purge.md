# Story 3.2: Configurable Retention & Purge

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the operator,
I want old transmission history to clean itself up automatically after a retention period I control,
so that the database doesn't grow forever and I'm not stuck with a fixed window.

## Acceptance Criteria

1. Given a retention period is set (default 30 days), When transmission records exceed that age, Then they are purged automatically on a recurring basis. [Source: epics.md#Story 3.2] [Source: mechanics.md#Retention and purge (CAP-8) — "Transmission history is purged once it exceeds a retention window. Default window: 30 days"] [Source: ARCHITECTURE-SPINE.md#Capability → Architecture Map — "CAP-8 Retention/purge | `domain/RetentionPolicy`, `worker/RetentionPurgeWorker` | AD-10; default retention 30 days"]
2. Given I change the retention period, When the new value is saved, Then it governs future purges only — already-deleted rows are not retroactively affected. [Source: epics.md#Story 3.2] [Source: mechanics.md#Retention and purge (CAP-8) — "changing it changes what gets purged on the next purge cycle, not retroactively re-purging already-deleted rows"]

**Scope boundary:** this story owns the **actual** configurable retention value and the worker that purges by it — Story 3.1 deliberately built the Dashboard's 30-day history window as a **hardcoded literal**, explicitly deferring "whether the dashboard should then read that live value instead... is Story 3.2's call to make when it exists" [Source: 3-1-delivery-dashboard.md#Dev Notes]. This story **does** make that call: Task 6 below updates `DashboardViewModel` to read the live `RetentionSetting` value instead of its hardcoded `HISTORY_WINDOW_DAYS = 30L` constant — otherwise raising the retention period would have no visible effect (the operator could retain 90 days of history but the dashboard would still only ever show 30), which defeats the AC's own stated purpose ("I'm not stuck with a fixed window"). Do **not** change the purge query's own age field, `TransmissionRepository.MAX_SEND_ATTEMPTS`, or any Story 2.2 send-dispatch logic — this story only adds a new setting, a new purge path, and rewires the Dashboard's window source. Per `EXPERIENCE.md`'s explicit IA note, this story is also what gives the previously-unbuilt Settings tab (bottom-nav index 3) its first and only real content — there is no other Settings functionality to build beyond the retention row.

### Review Findings

- [x] [Review][Patch] `RetentionPurgeWorker.doWork()` discarded the `AppResult<Int>` returned by `retentionPolicy.purgeExpired()` and unconditionally returned `Result.success()` unless an actual exception escaped — but every failure path in `RetentionRepositoryImpl`/`TransmissionRepositoryImpl` converts exceptions into `AppResult.Failure` rather than throwing, so a genuine DB failure during purge was silently reported as success and the `MAX_WORKER_RETRIES`/`Result.retry()` machinery could never fire for that entire failure class. Confirmed by all three review layers independently; had zero test coverage (`RetentionPurgeWorkerTest` only ever stubbed the `Success` case). Fixed by branching on the `AppResult` and added a dedicated `Failure`-path regression test. [worker/RetentionPurgeWorker.kt]
- [x] [Review][Patch] No bounds validation on `retentionDays` anywhere in the write path — `RetentionRepositoryImpl.setRetentionDays` accepted any `Int`, including 0 or negative, with the Settings dialog's own UI-level check as the *only* guard standing between it and a purge that would delete nearly all history on the very next daily run. Fixed by rejecting `days <= 0` with `AppResult.Failure(FailureReason.INVALID_INPUT)`, mirroring `ImageSelectionEngine`'s existing use of the same `FailureReason` for the same class of problem. [data/repository/RetentionRepositoryImpl.kt]
- [x] [Review][Patch] `DashboardViewModel` recomputed the CAP-8 cutoff (`now() - retentionDays days`) inline instead of delegating to `RetentionPolicy`, duplicating the exact formula AD-10 explicitly says must live in exactly one place ("Workers and ViewModels call them; they never reimplement the logic inline"). Caught by the Acceptance Auditor, who also noted Task 6's own literal instructions nudged the implementation toward this outcome — a genuine internal contradiction between Task 6's text and the AD-10 rule Task 3 itself quotes. Fixed by adding `RetentionPolicy.observeCutoff(): Flow<Instant>` and switching `DashboardViewModel` to depend on `RetentionPolicy` instead of `RetentionRepository` directly — this also let `DashboardViewModel` drop its own injectable `now` parameter entirely, since the clock now lives solely in `RetentionPolicy`. ## Dev Notes

- **`RetentionSetting` deliberately deviates from `ComplianceState`'s lazy-upsert pattern.** `ComplianceState` only gets its first real row once Setup completes (it needs the operator's nickname/city, which don't exist before that). `RetentionSetting` has no such dependency — the default (30) is known at compile time — so seeding it directly via migration SQL is simpler and avoids a null-until-Setup-completes window this setting doesn't actually need. Both are still singleton `id = 1` tables per ARCHITECTURE-SPINE.md's Consistency Conventions table.
- **Purge by `createdAt`, not `sentAt`.** `sentAt` is `null` for any row that never reached `SENT` (still `PENDING`, or `FAILED`), so a `sentAt`-based filter would let old failed/abandoned rows accumulate forever — exactly the unbounded-growth problem this story exists to prevent. `createdAt` is set at `enqueue()` time for every row regardless of outcome (see `TransmissionRepositoryImpl.enqueue`).
- **`RetentionPolicy` is the third and final domain service named in AD-10** (alongside `ImageSelectionEngine`, `ComplianceGate`) — architecture explicitly forbids the cutoff-date math living anywhere else. Resist the temptation to compute `now.minus(days, DAYS)` inline in the Worker "just this once."
- **`RetentionPurgeWorker`'s periodic interval (24h) is intentionally different from `SendWorker`'s (15min).** Don't copy `SendWorker`'s constants wholesale — the daily cadence is a deliberate, justified choice for a housekeeping task, not an oversight to "fix" toward consistency.
- **This story closes a decision Story 3.1 explicitly punted.** Its own Dev Notes said outright: "Story 3.2 introduces the actual configurable `RetentionSetting`; whether the dashboard should then read that live value instead of a hardcoded 30 is Story 3.2's call to make when it exists, not a decision to pre-empt here." Task 6 makes that call: yes, wire it up, because leaving the dashboard hardcoded after this story ships would make the retention *setting* invisible to the operator — they could set 90 days and never see more than 30 days of history, silently contradicting the feature's own premise.
- **No AC calls for a manual "purge now" button or a "last purged at" display.** The ACs only require automatic, recurring, future-only purging. Don't add operator-facing purge controls beyond the retention-days row itself — that would be scope creep against epics.md, the same discipline Story 3.1 held to for its own scope boundary.
- **No upper bound on retention days is specified anywhere** (`DESIGN.md`/`EXPERIENCE.md`/`mechanics.md` all only ever give the example "30 → 60" as illustrative, never a hard ceiling). Validate only that the entered value is a positive integer (`> 0`) — don't invent an arbitrary max the spec never asked for.
- **Numeric picker, not slider — this is explicit in two independent source documents** (`DESIGN.md#Components` and `EXPERIENCE.md#Component Patterns`), both citing the same reasoning: the value only ever needs coarse, occasional adjustment. A simple `AlertDialog` + `OutlinedTextField` (mirroring `ReceiverEditScreen`'s existing numeric-input convention) satisfies "numeric picker" without over-building a custom scrollable spinner widget for a low-frequency setting.
- **`Transmission`'s schema is unchanged by this story** — no new columns, no new migration touching that table. Only a brand-new `retention_settings` table is added (migration 6→7).
