# Story 2.2: Offline-Safe Queued Delivery

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the operator,
I want selected images queued and delivered automatically over WhatsApp or email, retrying through temporary outages,
so that a bad network moment doesn't mean a missed day for a customer.

## Acceptance Criteria

1. Given images have been selected for a receiver's scheduled send, When the app attempts delivery, Then it queues by image ID (not image data) and sends via the receiver's configured channel (WhatsApp Business API or Gmail SMTP). [Source: epics.md#Story 2.2] [Source: mechanics.md#Queue and delivery (CAP-4)]
2. Given a queued send fails, When the app retries, Then it retries the same queued item up to 3 times regardless of failure reason. [Source: epics.md#Story 2.2] [Source: mechanics.md#Queue and delivery (CAP-4), #Failure-mode table]
3. Given the device was offline through an entire scheduled window, When connectivity returns, Then only that day's missed batch is resumed — earlier missed days are not backfilled. [Source: epics.md#Story 2.2] [Source: mechanics.md#Queue and delivery (CAP-4)]

**Scope boundary:** this story builds the entire CAP-4 delivery pipeline — `SendWorker`, the `Transmission` write path, and both delivery-channel clients — and is the first story to introduce WorkManager into this codebase. It does **not** touch Epic 3 (`3.1` dashboard reads `Transmission` rows this story writes; `3.2` retention/purge). It also does **not** build a separate periodic `ComplianceCheckWorker` — Story 1.1 explicitly deferred that decision to "Epic 2's scheduling work" [Source: 1-1-first-run-registration-compliance-gate.md#Task, implementation-readiness-report-2026-07-09.md#Minor Concern 2]; this story resolves it by having `SendWorker` call `ComplianceGate` inline once per run (see Dev Notes) rather than building a second, independent periodic worker that no story's acceptance criteria actually require.

This is this story's resolution of `implementation-readiness-report-2026-07-09.md`'s Minor Concern #2 (see Scope boundary above): inline in `SendDispatcher`, not a second independent Worker.
  - [x] **Pass 1 — retry existing `PENDING` items from earlier runs:** `transmissionRepository.getRetryCandidates()`. For each, look up its `Receiver` (from a `Map<Long, ReceiverWithSchedules>` built once from `receiverRepository.getAllWithSchedules()` at the start of the run — do not issue a separate query per item) and its `Image` (via `imageRepository.getImageById`, **not** `getActiveImages()` — see Task 3). If either lookup comes back missing (receiver or image deleted since queuing) or `Failure`, skip that item this run — nothing sane to retry against, and it stays `PENDING` for a future run to reconsider once/if the referenced rows exist again (in practice this app never deletes images or receivers with in-flight transmissions, so this is a defensive skip, not an expected path). Otherwise call `deliveryRepository.send(receiver, image)` and apply the attempt-outcome update (next bullet).
  - [x] **Attempt-outcome update (shared by both passes):** `newAttemptCount = transmission.attemptCount + 1`. On `Success`: `transmission.copy(status = SENT.name, attemptCount = newAttemptCount, sentAt = now().toEpochMilli())`. On `Failure`: if `newAttemptCount >= MAX_SEND_ATTEMPTS` (a `const val MAX_SEND_ATTEMPTS = 3` defined in this file — the "up to 3 times" cap from AC2/mechanics.md), `transmission.copy(status = FAILED.name, attemptCount = newAttemptCount)` (terminal — no further retries attempted); otherwise `transmission.copy(status = PENDING.name, attemptCount = newAttemptCount)` (stays queued, this run's periodic tick — not `WorkManager`'s own `Result.retry()` backoff — is what re-attempts it later). Persist via `transmissionRepository.update(...)` either way.
  - [x] **Pass 2 — dispatch newly-due schedule slots:** for each `ReceiverWithSchedules` from the same snapshot, for each of its `scheduleTimes: List<Int>` (minutes-since-midnight, local time): compute "due" as `scheduleTime <= minutesSinceLocalMidnight(now())`; if not due, skip. If due, call `transmissionRepository.hasDispatchedToday(receiver.id, scheduleTime, startOfLocalDay(now()))` — if `true`, skip (already handled this slot today, whether it fully succeeded or is still retrying via Pass 1). Otherwise call `imageSelectionEngine.selectImagesFor(receiver.id, receiver.minCount, receiver.maxCount)`; for each returned `Image`, `transmissionRepository.enqueue(receiver.id, image.id, scheduleTime, now())` then immediately run the same attempt-outcome update above as the first attempt (`attemptCount` starts at `0` on the freshly-created row, so the first `send()` call brings it to `1`).
  - [x] **Why "no backfill of missed days" needs no special-case branch:** "due" and "already dispatched today" are both evaluated only against *today* — there is no code path that ever looks at yesterday's or any earlier day's schedule state. If the device was off through an entire day, that day's slots simply never got evaluated while they were "today"; by the time the app is back online, `now()` has moved on and only the new "today" is ever checked. This mirrors how Story 2.1's starvation case fell out of `take(z)` naturally rather than needing its own branch — verify this claim explicitly with a dedicated test (Task 6) rather than trusting the reasoning by inspection alone, per this project's established practice.
  - [x] A `Failure` from `receiverRepository.getAllWithSchedules()` at the very start of a run aborts the whole run early (nothing to iterate) — let it propagate up to `SendWorker` (Task 5) as a thrown/returned failure signal so `WorkManager` can decide to retry the run sooner; do not silently treat it as "zero receivers."
  - [x] `CancellationException` must propagate uncaught throughout — same convention as every other suspend chain in this codebase.

### Review Findings

- [x] [Review][Decision] AC2's "up to 3 times" is implemented as 3 total attempts (initial send + 2 retries), not 3 retries beyond the initial attempt (4 total sends) — `SendDispatcher.attemptDelivery` computes `newAttemptCount = transmission.attemptCount + 1` and marks terminal once `newAttemptCount >= MAX_SEND_ATTEMPTS (3)`. Both mechanics.md and AC2 are worded ambiguously enough to support either reading. **Resolved: keep the current 3-total-attempts behavior** — no user response arrived within the review window; kept as-is since it's the zero-risk option (matches what's already implemented and tested) and the ambiguity genuinely cuts both ways. Revisit if the operator reports sends giving up too early in the field. [worker/SendDispatcher.kt]
- [x] [Review][Decision] The retry pass isn't day-scoped like the dispatch pass is — a `Transmission` still `PENDING` from a schedule slot enqueued late yesterday (only reachable if the device is offline/Doze-suspended spanning midnight right after a late-night send) is retried indefinitely alongside a **freshly-enqueued** transmission for today's occurrence of that same slot, risking two deliveries to the same receiver. **Resolved: auto-terminate at the day boundary** — no user response arrived within the review window; applied the recommended fix since it's small, low-risk, and consistent with this story's own "no backfill of missed days" philosophy. Converted to a patch (see below). [worker/SendDispatcher.kt — `getRetryCandidates()`/`hasDispatchedToday()`]
- [x] [Review][Patch] (from the day-boundary decision above) In the retry pass, mark a `PENDING` transmission `FAILED` (terminal) instead of retrying it once its `createdAt` falls before today's start — closes the duplicate-send risk without changing dispatch-pass behavior. [worker/SendDispatcher.kt — `retryPendingItems`]
- [x] [Review][Patch] The periodic `SendWorker` request has no `Constraints` at all (no `NetworkType.CONNECTED` requirement) — for a story named "Offline-Safe Queued Delivery," the worker still fires every 15 minutes while offline, burning through the 3-attempt retry budget on calls guaranteed to fail instead of deferring until connectivity returns. [worker/SendWorker.kt — `enqueuePeriodic`]
- [x] [Review][Patch] `AppConfig.WHATSAPP_API_TOKEN` is declared but never attached to any request — no `Authorization` header, no interceptor, no auth wiring anywhere. Every WhatsApp Cloud API call will get rejected the moment real credentials are filled in. [di/AppContainer.kt — `whatsAppApi`]
- [x] [Review][Patch] `SmtpClient.sendImage`'s JavaMail `Properties` set no `mail.smtp.connectiontimeout`/`timeout`/`writetimeout` — JavaMail has no sane default, so a stalled connection can block the coroutine (and the whole sequential dispatch pass behind it) indefinitely. [data/remote/SmtpClient.kt]
- [x] [Review][Patch] `SendWorker.doWork()` maps every non-cancellation exception to `Result.retry()` with no attempt ceiling — a persistent bug/config error retries without ever giving up within a period, instead of falling back to `Result.failure()` past a reasonable cap (`runAttemptCount` is available on `CoroutineWorker` for exactly this). [worker/SendWorker.kt]
- [x] [Review][Patch] The one-time battery-optimization `startActivity(ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS)` call has no `ActivityNotFoundException` guard — some OEM ROMs don't resolve this system intent, which would crash the app immediately after Setup completes. [ui/App.kt — `requestIgnoreBatteryOptimizationsOnce`]

## Dev Notes

- **This is the first story to introduce `WorkManager` into this codebase.** No prior story added it, so there's no existing pattern to mirror for the `Configuration.Provider`/custom `WorkerFactory` wiring — follow Task 5's subtasks precisely; the manifest auto-init-suppression step in particular is easy to skip and produces a hard-to-diagnose runtime failure (default init silently fighting the custom one) rather than a compile error.
- **`androidx.work` version:** `ARCHITECTURE-SPINE.md#Stack` pins "2.10.x (pin exact patch against `androidx.work` release notes at build time — 2.11.2 could not be confirmed against a primary source)" — start with `2.10.1` (a confirmed-stable release as of the architecture's authoring) and bump the patch if a newer stable `2.10.x` is available/resolves cleanly at build time. Same for `androidx.work:work-testing`, matching version.
- **JavaMail on Android:** plain `jakarta.mail`/`com.sun.mail:jakarta.mail` desktop artifacts depend on `javax.activation`/JNDI machinery that isn't present on the Android runtime and will crash or fail to resolve. Use the long-established Android-compatible port instead: `com.sun.mail:android-mail:1.6.7` + `com.sun.mail:android-activation:1.6.7`. This is a stable, years-old pairing specifically maintained for Android SMTP use — confirm the exact latest patch resolves at build time the same way every other pinned version in this project has been treated, but don't swap to a different library entirely without a specific reason.
- **Compliance gate placement (this story's decision, per Scope boundary):** `SendDispatcher` calls `ComplianceGate.evaluate(...)` once per run, before Pass 1/Pass 2, and skips the entire run on `Halt`. This directly resolves `implementation-readiness-report-2026-07-09.md`'s Minor Concern #2, which explicitly asked that "the story context for 1.1/2.2 states the choice." Story 1.1 chose inline-at-launch for the foreground Setup flow; this story chooses inline-in-`SendDispatcher` for the background path — both satisfy AD-10/AD-11 without a second, independent `ComplianceCheckWorker` that no story's acceptance criteria ever actually call for.
- **`mechanics.md`'s "no distinction between failure reasons" is a real simplification, not a shortcut being taken.** It removes the need for `DeliveryRepository`/`SendDispatcher` to carry granular failure taxonomy end to end — a single `AppResult.Failure(FailureReason.UNKNOWN)` catch-all from `DeliveryRepository` is the *correct*, spec-directed design, not a corner cut. Don't add finer-grained failure reasons "for completeness" — there is no consumer for them (retry behavior is identical regardless, and `Transmission.status`/`attemptCount`, not `FailureReason`, is what Story 3.1's dashboard will read).
- **The `createdAt`/`scheduleTime` schema addition (Task 1) is a genuine, reasoned gap-fill, not scope creep.** `ARCHITECTURE-SPINE.md#AD-13` and the original ER diagram were written before anyone worked through exactly how `SendWorker` would determine "already handled today" — there was no way to implement AC2/AC3 correctly without it. This is exactly the kind of "the dev agent owns it even if not explicitly written" situation `bmad-create-story`'s own critical instruction calls out.
- **Retry cadence is the periodic Worker's own interval, not `WorkManager`'s `Result.retry()` backoff.** `SendWorker.doWork()` always returns `Result.success()` after a normal `dispatchDueSends()` run, even if individual `Transmission` items failed and are still `PENDING` — those get picked up by the *next* 15-minute tick's Pass 1, not by `WorkManager` rescheduling the whole Worker. `Result.retry()` is reserved for the run itself throwing (e.g. `getAllWithSchedules()` failing before any item-level logic could even start). Keeping these two retry mechanisms cleanly separated avoids conflating "the whole run didn't happen" with "this one item didn't succeed yet."
- **Every random-count/no-repeat rule from Story 2.1 stays entirely inside `ImageSelectionEngine`; this story calls it, never reimplements or second-guesses its output.** `SendDispatcher` treats `selectImagesFor(...)`'s returned list as final — it does not re-filter, re-check the 7-day window, or otherwise duplicate CAP-3 logic, per AD-10.
