# Story 3.1: Delivery Dashboard

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As the operator,
I want to see, per receiver, what was sent and when, going back at least 30 days,
so that I can resolve a "I didn't get my images" dispute.

## Acceptance Criteria

1. Given transmissions have occurred, When I open the dashboard for a receiver, Then I see a timestamped list of every image sent to them within the retention window. [Source: epics.md#Story 3.1] [Source: mechanics.md#Retention and purge (CAP-8) — "Default window: 30 days, matching the dashboard's display range"]
2. And I can switch to a different receiver to see their own history independently. [Source: epics.md#Story 3.1] [Source: EXPERIENCE.md#Component Patterns — "Dashboard receiver picker... switching it reloads the list below without leaving the screen"]

**Scope boundary:** the "retention window" this story displays is a **hardcoded 30-day lookback** (`Instant.now().minus(30, ChronoUnit.DAYS)`), matching mechanics.md's stated default exactly — it does **not** read from a configurable `RetentionSetting` table. That table, the operator-facing control to change the retention period, and the actual purge worker that deletes old rows are all Story 3.2's job ("Configurable Retention & Purge"). Do not build `RetentionSetting`, a settings UI, or any purge logic here. This story is also **read-only** — it only queries `Transmission` rows Story 2.2's `SendDispatcher` already writes; it does not write to `transmissions` at all. Per EXPERIENCE.md's own explicit note, failed/retrying sends are **not** shown as a visible "failed" state — a slot with no successful `SENT` row simply has no entry in the list; do not invent a failure indicator no story's ACs call for.

## Dev Notes

- **This is the first story to read `Transmission` data for display rather than to write it or feed it into CAP-3's selection logic.** `ImageSelectionEngine` (2.1) and `SendDispatcher` (2.2) are the only writers; this story adds a third, purely read-only consumer. Do not add any write path here.
- **The 30-day hardcoded window is deliberate, not a shortcut.** `mechanics.md` explicitly ties the default retention window to "the dashboard's display range" — both are 30 days by design. Story 3.2 introduces the actual configurable `RetentionSetting`; whether the dashboard should then read that live value instead of a hardcoded 30 is Story 3.2's call to make when it exists, not a decision to pre-empt here.
- **No "failed send" UI anywhere, by explicit design.** `EXPERIENCE.md` flags this itself as an `[ASSUMPTION]`: no story currently calls for a visible failure indicator, and inventing one here would be scope creep against epics.md. A `PENDING`/`FAILED` transmission simply produces no dashboard row for that slot.
- **Reuse `ImageRepository.resolveFile(filePath)` for thumbnails — don't reintroduce file-path math.** `ImageLibraryScreen`'s `ImageGridItem` already established the `AsyncImage(model = file, ...)` pattern against a resolved `File`; the `DeliveryRecord.image.filePath` needs the exact same resolution, through the exact same method.
- **`success-muted` (`#128C7E`) is the same hex as `colors.primary`** — this is intentional per `DESIGN.md` (a "quiet" reuse, not a new named color needing its own `Color(...)` constant if `MaterialTheme.colorScheme.primary` already resolves to it; check `ui/theme/Theme.kt`'s color scheme definition before adding a redundant new constant).
- **Null-vs-empty StateFlow discipline continues here.** Both `receivers` and `history` follow the same `StateFlow<List<X>?>` convention as `ReceiversViewModel.receivers` (Story 1.3's own review-fixed lesson) — `null` means "haven't loaded yet," empty list means "loaded, and there's genuinely nothing here." Do not collapse these with `?.isNullOrEmpty()` anywhere in the UI layer, the exact regression Story 1.3's second review round caught and fixed.

### Previous Story Intelligence

- **No Story 3.0 exists** (this is the first story in Epic 3) — there is no same-epic predecessor to inherit review feedback from. The relevant prior-work lessons instead come from Epic 1/2's own review history, cited inline above: Story 1.3's null-vs-empty `StateFlow` fix, Story 2.1/2.2's bound-SQL-parameter convention, and Story 1.3's `ReceiverRepositoryImplTest` real-DB-for-`@Relation`-queries precedent.
